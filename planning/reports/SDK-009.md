## Summary

Implemented registration status polling so SDK users and CLI operators can track a funded-account registration from `pending` through a terminal state (`registered`/`active` or `failed`).

This work extends the `RegisterClient` added in SDK-008 with:

- a single-shot status check against `GET /api/registration-status?hl_address=...`
- a configurable polling loop that waits for a terminal status
- an `on_status` callback for callers to observe intermediate states
- a `RegistrationPollTimeoutError` for bounded polling
- a full CLI `register status` command with `--poll/--no-poll`, `--json`, `--interval`, and `--timeout` options

During implementation, the status polling interface was adapted from the original sprint plan based on alignment with the actual backend contract: the backend identifies registrations by `hl_address`, not by `registration_id`.

## Details

### 1. Status check

`check_status_async(hl_address)` performs a single `GET /api/registration-status?hl_address=...` and returns a `RegistrationStatus`. Wallet validation is applied before the request. Backend error responses (400, 502, etc.) surface as `RegistrationError`.

### 2. Polling loop

`poll_until_complete_async(hl_address, *, interval_seconds, timeout_seconds, on_status)` repeatedly calls `check_status_async` until the returned status is in `TERMINAL_STATUSES` (`registered`, `active`, `failed`, `eliminated`).

- `interval_seconds` controls the delay between polls (default 5s)
- `timeout_seconds` bounds total polling time (default 300s)
- `on_status` is invoked with each `RegistrationStatus` for progress reporting
- Raises `RegistrationPollTimeoutError` on timeout, preserving the last observed status

### 3. Terminal status classification

The `RegistrationStatus` model now includes two computed properties:

- `is_terminal` — `True` when `status` is in `{"registered", "active", "failed", "eliminated"}`
- `is_success` — `True` when `status` is in `{"registered", "active"}`

These cover both the hyperscaled app's status vocabulary (`registered`, `pending`) and the vanta-network validator's vocabulary (`active`, `eliminated`).

### 4. CLI `register status` command

```bash
# Poll until terminal (default)
hyperscaled register status --hl-wallet 0x...

# Single check, no polling
hyperscaled register status --hl-wallet 0x... --no-poll

# JSON output
hyperscaled register status --hl-wallet 0x... --no-poll --json

# Custom interval and timeout
hyperscaled register status --hl-wallet 0x... --interval 10 --timeout 600
```

Exit codes:
- `0` — terminal success (`registered` or `active`)
- `1` — terminal failure (`failed`) or backend error
- `2` — polling timeout

### 5. Sprint plan adaptations

| Sprint Plan | Implementation | Reason |
|-------------|---------------|--------|
| `check_status(registration_id)` | `check_status(hl_address)` | Backend identifies registrations by HL address; `registration_id` is not returned by the app |
| Persist `funded_account_id` on success | Not implemented | Backend does not return `funded_account_id`; the config field exists and is ready for when it does |
| Expected response: `registration_id`, `funded_account_id`, `account_size` | Actual response: `status`, `hl_address` | SDK models the actual backend contract rather than the planned shape |

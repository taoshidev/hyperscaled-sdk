## Summary

Implemented the account-management surface for configuring the SDK against a Hyperliquid wallet and checking whether that wallet meets the minimum funded-account requirement ($1,000). This mirrors the balance-gating behavior already present in the Chrome extension, but exposes it in both the Python SDK and CLI.

- `client.account.setup(wallet_address)` validates and persists the HL wallet to local config
- `client.account.check_balance()` queries the Hyperliquid info API and returns balance, meets_minimum, and minimum_required
- `client.account.watch_balance(callback)` continuously polls balance and invokes the callback on each update
- `hyperscaled account check` and `hyperscaled account check --json` CLI commands
- `BalanceStatus` Pydantic model and `MINIMUM_BALANCE` constant for reuse across SDK surfaces
- 33 mock-based tests covering healthy balance, insufficient balance, edge cases, missing wallet config, and Hyperliquid API failure handling

## Details

### 1. `BalanceStatus` model

New Pydantic model representing the result of a balance check:

```python
class BalanceStatus(BaseModel):
    balance: Decimal
    meets_minimum: bool
    minimum_required: Decimal = MINIMUM_BALANCE  # Decimal("1000.00")
```

`MINIMUM_BALANCE` is defined as a module-level constant in `models/account.py` so it can be reused by SDK-008 pre-trade gating and any future balance-related logic without duplicating the threshold.

### 2. `AccountClient.setup()`

Validates the wallet address using the shared `is_valid_hl_address()` helper from SDK-006, then persists it to local config:

```python
async def setup_async(self, wallet_address: str) -> None:
    if not self.validate_wallet(wallet_address):
        raise ValueError(...)
    self._client.config.set_value("wallet.hl_address", wallet_address)
    self._client.config.save()
```

Raises `ValueError` on invalid input so the caller (CLI or script) can decide how to surface the error.

### 3. `AccountClient.check_balance()`

Queries the Hyperliquid info API (`clearinghouseState`) for the wallet's USDC account equity:

```python
async def check_balance_async(self, wallet_address: str | None = None) -> BalanceStatus:
    resolved = self._resolve_wallet(wallet_address)
    balance = await self._fetch_hl_balance(resolved)
    return BalanceStatus(balance=balance, meets_minimum=balance >= MINIMUM_BALANCE)
```

Behavior:

- Uses the configured HL wallet unless one is explicitly overridden
- Returns a structured `BalanceStatus` with the exact balance, a boolean `meets_minimum`, and the threshold
- Raises `HyperscaledError` for missing wallet config, HTTP errors, network failures, and unexpected response shapes

### 4. `AccountClient.watch_balance()`

Continuously polls balance and invokes a callback on each update:

```python
async def watch_balance(
    self,
    callback: Callable[[BalanceStatus], Any | Awaitable[Any]],
    *,
    wallet_address: str | None = None,
    poll_interval: float = 5.0,
) -> None:
```

Supports both sync and async callbacks. Runs until the task is cancelled via `asyncio.CancelledError`.

### 5. CLI commands

`hyperscaled account check` prints a human-readable pass/fail result:

```
Balance:          $1,250.42
Minimum required: $1,000.00
Status:           PASS
```

`hyperscaled account check --json` outputs machine-readable JSON:

```json
{
  "balance": "1250.42",
  "meets_minimum": true,
  "minimum_required": "1000.00"
}
```

`hyperscaled account check --wallet 0x...` allows overriding the configured wallet for a one-off check.

### 6. Reusability for downstream tickets

The `check_balance_async()` method is designed to be called by SDK-008 (funded account purchase) and future trade submission flows. When the balance is below the minimum, those flows should raise `InsufficientBalanceError` (already defined in `exceptions.py`) using the `BalanceStatus` result rather than duplicating the Hyperliquid API call or threshold logic.
